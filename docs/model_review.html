Executive summary

The model is broadly sound and internally disciplined. The canonical fractional-ETR convention is respected on the live paths, the noncompliance (a)/(b)/(c) decomposition is implemented coherently, and the calibration harness reproduces its golden fixtures bit-for-bit. The large majority of findings are low-severity consistency, documentation, or defensive-coding gaps that are latent on current data rather than active wrong numbers.

There is one critical bug that corrupts a headline output. The temporary-tariff IRFs are shifted using the permanent-shock onset constant, delaying every temporary-tariff macro response by exactly three quarters — distorting the flagship 2026 Q4-over-Q4 GDP, unemployment, PCE, and fed-funds figures for every scenario with a temp window (including the live 2026-04-06 family). A second issue in the same module compounds it by double-counting overlapping temp IRFs.

The biggest recurring theme is a noncompliance/substitution inconsistency in the time-varying revenue path: the per-fiscal-year revenue trajectory is anchored to a post-noncompliance (b) GTAP level but scaled by a pre-noncompliance (a) time profile (found independently by two reviewers), and a related FY-window edge carries the first tariff regime into pre-policy days, overstating FY2026 revenue. A second theme is silent-drop / coercion fragility (inner_join on PCE keys, missing nrow>0 guards, Date-vs-character filters) — dormant today, but each a house-rule violation that exists precisely to surface the data problems they would otherwise mask.

Trust assessment

  • Price, ETR, sector, foreign-GDP, distribution levels — trustworthy at current parameters.
  • Macro 2026 Q4-Q4 numbers — do not trust until the temp-IRF onset bug (and the overlap double-count) are fixed.
  • Conventional revenue — slightly overstated in FY2026; bounded inter-year profile bias under active noncompliance. The 10-year total is affected but not grossly.
  • Everything else is cosmetic or maintainability.

1 · The critical bug (verified directly)

critical Temp IRF shifted by the permanent-shock onset constant

src/05a_usmm_surrogate.R:337-347

The perm and temp IRF families have different native onset quarters. I confirmed this against the raw IRF data:

  • perm10_fed.csv — first material response at 2025Q2 = index 6 (= USMM_SHOCK_QUARTER). ✓
  • temp10_fed.csv — first material response at 2026Q1 = index 9 (2025Q4 holds only a -1e-14 numerical zero). 3 quarters later.

construct_usmm_response() computes a single onset_shift = comp$onset_q − USMM_SHOCK_QUARTER (=6) and applies it to both the shifted-perm path and the temp path via shift_irf(temp_irf[[var]], onset_shift). For the perm path this correctly re-centers a 2025Q2 IRF; for the temp path it adds the 3-quarter gap on top of the IRF's already-2026Q1 native onset.

Worked through the live 2026-04-06 config (temp_component_dates=[2026-02-24, 2026-07-24]): comp1 onset 2026-02-24 → onset_q=9 → onset_shift=3, pushing the temp IRF from index 9 (2026Q1) to index 12 (2026Q4); comp2 is pushed into 2027Q1. Every temporary-tariff response moves ~3 quarters too late — largely out of the 2026 window that defines the flagship 2026 Q4-over-Q4 numbers. Static and shifted-perm-only scenarios are unaffected.

Fix — introduce USMM_TEMP_SHOCK_QUARTER <- 9 and shift temp components by comp$onset_q − 9, keeping perm components on USMM_SHOCK_QUARTER. Add an assertion that each IRF family's first-nonzero quarter matches its assumed native onset, so a future IRF re-extraction can't silently desync. (refund_fed.csv is correctly added with no shift.)

2 · Highest-impact medium findings

medium Overlapping temp IRFs summed, double-applying reversion dynamics

src/05a_usmm_surrogate.R:258-262

The match predicate use_temp <- any(onset >= temp_pairs$onset & offset <= temp_pairs$offset) flags every telescoping component fully contained in the temp window. For 2026-04-06, both interior segments [02-24→04-06] and [04-06→07-24] qualify, so two separate deltas are each fed through the temp IRF. Unlike the shifted-perm path (which differences on/off to telescope correctly), the temp branch has no offset truncation — and the temp IRF already embeds a full rise-and-revert path, so summing two same-sign overlapping episodes compounds the reversion dynamics for what is conceptually one temporary regime with a mid-window step-up. Affects 4 live scenarios. Best fixed together with the critical bug — same function.

Fix — decide whether a temp window maps to one temp-IRF impulse (sized to the window's peak delta, placed at window onset) or one IRF per interior segment; if the latter, document why summing overlapping un-truncated temp IRFs is correct. Otherwise collapse the window to one temp component.

medium FY-weighted ETR carries the first regime into pre-policy days

src/04_calculate_revenue.R:26-27, 64-71

compute_fy_weighted_etr_increase's docstring says "Before the first date: etr_increase = 0", but the implementation uses increases[1] (the first regime's tariffed rate) when no schedule date precedes the interval start. The verifier corrected the original note: produced outputs have earliest date 2026-01-01, so FY2026 (2025-10-01 → 2026-09-30) is inside the budget window. When the first schedule date (2026-01-01) falls after fy_start (2025-10-01), the pre-policy Oct–Dec 2025 quarter is treated as fully tariffed. Reproduced: with first date 2026-01-01 @ 0.10 and 2026-02-20 @ 0.14, FY2026 etr_increase = 0.1244 vs the correct 0.0992 — a 2.52pp overstatement flowing into the 10-year headline total with no clamp.

Fix — set active_etr <- 0 when length(active_idx)==0. Baseline subtraction already accounts for pre-policy collections, so 0 is the economically correct pre-policy value. Reconcile the docstring.

medium Time-varying revenue: (b) level anchor scaled by an (a) time profile

src/02_calculate_etr.R:438-461  (two reviewers converged)

For time-varying scenarios the per-FY etr_increase is mtax_etr_increase * (weighted_etr / ref_etr). The level anchor traces to a GTAP run shocked with the rate-(b) eta′-adjusted reference-date rates, but the scaling ratio comes from inputs$etr_matrix_by_date — the rate-(a) applied-statutory delta. So the revenue level is post-noncompliance (b) while its time profile is pre-noncompliance (a). Internally contradictory: the same module hands USMM a (b) time profile but hands revenue an (a) one off the same (b) level. Bias is exactly zero at the reference date and when noncompliance is inactive; it bites only when eta′ composition shifts across dates. Diluted by FY day-weighting → medium.

Fix — when inputs$noncompliance_active, build the scaling ratio from the (b) per-date series (per_date_etrs_b / ref_b_etr, already computed at 02_calculate_etr.R:484-492 for USMM). Trivial — the inputs already exist.

medium uncertain — needs owner domestic_pricing=0.5 markup vs the overhaul memo

src/io_price_model.R:763-767

The applied scaling is 1 + domestic_pricing − usd_offset = 1 + 0.5 − 0.174 = 1.326, multiplying every pre-sub and PE-post-sub commodity price effect up ~33% before the USD offset. This contradicts price_model_overhaul_memo.md, which presents the new formula as ×(1 − USD offset) and argues the old 0.50 multiplier "is hard to defend." Key correction: the in-file docstring, @param block, config comment, and runtime log are all consistent with the code; git history (3f24056 shipped (1−usd_offset); 6297b8e deliberately re-added domestic_pricing after) strongly indicates the code is intentional and the memo is stale. The drift lives only in out-of-file notes (memo, MEMORY.md). But this is a methodology decision: if 0.5 is not the intended central case, headline price effects are ~33% high.

Fix — confirm intent. If intentional, update price_model_overhaul_memo.md + MEMORY.md to document/justify the retained markup. If not, set domestic_pricing=0.

3 · Cross-cutting themes

  1. (a) vs (b) discipline is the model's main coherence risk. The pipeline routes (b) to GTAP/USMM/prices correctly, but the time-varying revenue path mixes a (b) level with an (a) profile (#4), and the FY-window edge (#3) overstates pre-policy days. Fixing both — via one noncompliance_active branch in the revenue per-date scaling — would make revenue use the same (b) concept USMM already uses.
  2. IRF onset assumptions are fragile and unasserted. #1 and #2 both stem from assuming all IRF families share one onset/placement convention. A family-level "first-nonzero quarter matches assumed native onset" assertion would have caught #1 at re-extraction time.
  3. Silent-drop / coercion as a recurring anti-pattern. inner_join on PCE keys, missing nrow>0 guard on the (b) loader, message()-not-stop() on unmatched tau_M, and Date-vs-character filters are all dormant but each contradicts the explicit house rule that missing values must surface. Cheap to harden; the rules exist precisely to catch the refactor's data-shape changes.
  4. The code is more trustworthy than the prose right now. 30441 appears inline duplicating a named CBO parameter; the price-formula narrative drifted across memo/MEMORY/brief while the code stayed consistent; bea_io_level's default is stated four ways. Narrative docs need a reconciliation pass.
  5. Reference-date approximations stretched across time-varying schedules. R_alpha (#14) and the (a)-profile scaling (#4) share a root: GTAP solves only at the reference date, and several quantities from that one solve are applied uniformly across a multi-date schedule. Worth a documented statement of which quantities are reference-date approximations and why that's acceptable.
